# Roadmap: Stake Horse Racing Advisor Bot

## Overview

Three phases deliver the full v1 pipeline. Phase 1 builds the foundation (Docker service skeleton, FSM state machine, paste ingestion, LLM-based parsing with odds normalization, bankroll management, audit trail). Phase 2 adds the intelligence layer (web research, deterministic EV/Kelly math engine, AI analysis with betting labels, portfolio risk caps). Phase 3 closes the loop (result evaluation with placed/tracked distinction, memory-distilled AI reflection, P&L stats, drawdown circuit breaker). Each pipeline step is an independent async function callable from both the sequential runner and a future LangGraph agent loop. Agent mode (AGENT-01) is explicitly v2 and not part of this roadmap.

**Architectural constraint (ARCH-01):** All numerical calculations (overround, no-vig probability, EV, Kelly fraction, USDT amounts) are performed by deterministic Python functions — not LLM. This applies to all phases.

## Phase Details

### Phase 1: Foundation and Parser
**Goal**: User can paste raw Stake.com race text into Telegram and receive a confirmed, structured race summary ready for analysis — with odds normalized to decimal, scratched runners excluded, bankroll known and tracked, and every run logged to audit trail.
**Depends on**: Nothing (first phase)
**Requirements**: INPUT-01, INPUT-02, PARSE-01, PARSE-02, PARSE-03, PARSE-04, PARSE-05, PARSE-06, BANK-01, BANK-02, BANK-03, BANK-04, BANK-05, PIPELINE-01, PIPELINE-02, PIPELINE-03, PIPELINE-04, PIPELINE-05, AUDIT-01
**Success Criteria** (what must be TRUE):
  1. User pastes raw Stake.com text (or sends .txt file) and the bot responds with a structured race summary (runners with status, decimal odds, implied prob, overround, track, surface, place terms) within one message
  2. User is shown the parsed race summary and must confirm it before the pipeline continues — bot does not auto-proceed
  3. Odds normalization is unit-tested: decimal/fractional/American inputs all produce correct decimal output and correct overround
  4. Scratched runners are excluded from output and flagged; overround recalculated without them
  5. Bot always shows current USDT balance in every response header; balance persists across bot restarts; if no balance found in paste and none in DB, bot asks explicitly
  6. Every pipeline run writes a JSON-lines entry to the audit log covering: raw input, parsed output, and user confirmation/changes
  7. User can /cancel at any time; starting a new paste while pipeline is running triggers a duplicate warning

### Phase 2: EV Engine and Analysis
**Goal**: Given a confirmed parsed race, the bot runs web research on each runner, computes overround/EV/Kelly via deterministic Python functions, produces AI analysis with betting-relevant labels, applies portfolio risk caps and uncertainty discount, and delivers exact USDT bet amounts — or issues a skip signal when the margin makes betting structurally unprofitable.
**Depends on**: Phase 1
**Requirements**: SEARCH-01, SEARCH-02, ANALYSIS-01, ANALYSIS-02, ANALYSIS-03, ANALYSIS-04, ANALYSIS-05, BET-01, BET-02, BET-03, BET-04, BET-05, BET-06, BET-07
**Success Criteria** (what must be TRUE):
  1. Bot retrieves recent form, trainer stats, and expert opinions for each runner via SearXNG (or OpenRouter online model, configurable via env)
  2. AI labels each runner with a betting-relevant category: `highest_win_probability`, `best_value`, `best_place_candidate`, or `no_bet`
  3. Overround, no-vig probabilities, EV, Kelly fractions, and USDT amounts are all produced by Python functions — not generated by LLM; values are deterministic given the same inputs
  4. When bookmaker margin exceeds 15% (configurable), bot issues explicit skip signal with margin explanation instead of a bet recommendation
  5. Final output respects all sizing constraints: quarter-Kelly default, 3% per-bet hard cap, 5% total race exposure cap, max 2 win bets per race; when research data is sparse for a runner, sizing is halved and flagged
  6. If research finds significantly different odds for a runner in external sources, a market discrepancy note appears in the analysis output
  7. Audit log entry updated with recommendation data for this run

### Phase 3: Results, Reflection and Stats
**Goal**: User can submit a race result back to the bot; the system evaluates every recommended bet (distinguishing placed from tracked), updates P&L, writes a memory-distilled AI reflection, makes statistics visible on demand, and activates a drawdown circuit breaker if the bank drops 20% from peak.
**Depends on**: Phase 2
**Requirements**: RESULT-01, RESULT-02, RESULT-03, TRACK-01, REFLECT-01, REFLECT-02, REFLECT-03, STATS-01, RISK-01
**Success Criteria** (what must be TRUE):
  1. User can submit a race result as flexible text (e.g. "3,5,11,12" or "horse name won"); bot parses it and asks for clarification if ambiguous
  2. User marks the recommendation as `placed` or `tracked`; P&L stats count only `placed` bets; model quality metrics count both
  3. System evaluates each recommended bet against the actual result and displays the P&L breakdown
  4. AI reflection entry written to `mindset.md` after each result — must include what went wrong even in winning bets; immediately followed by extraction of one structured lesson (error tag + rule sentence) appended to the lessons store
  5. Next race's analysis prompt receives top-5 extracted rules + last 3 failure modes — not raw reflection text
  6. User can request P&L stats (total, by period, win rate, ROI) for `placed` bets at any time
  7. If bankroll drops ≥20% from its recorded peak, all subsequent recommendations are automatically issued as "SKIP (drawdown protection)" until user explicitly unlocks
  8. Audit log entry updated with result and P&L for this run
