**Create `services/stake/pipeline/research/agent.py`** implementing the two-tier pattern per D-02:

The senior orchestrator (expensive model, `AnalysisSettings.model` = gemini-pro) creates a research plan and synthesizes results. Cheap sub-agents (`ResearchSettings.model` = flash-lite) execute individual searches.

1. **`build_search_sub_agent(settings)`** factory function:
   - Creates a lightweight `create_react_agent` using `ResearchSettings.model` (flash-lite) — the CHEAP model
   - Uses `ChatOpenAI` with `openai_api_base="https://openrouter.ai/api/v1"`, `openai_api_key=settings.openrouter_api_key`, `model=settings.research.model`, `temperature=settings.research.temperature`, `max_tokens=settings.research.max_tokens`
   - If `settings.research.provider == "online"`: tools = `[online_model_search]`, add `extra_body={"plugins": [{"id": "web"}]}` to ChatOpenAI
   - If `settings.research.provider == "searxng"`: tools = `[searxng_search]`, no extra_body
   - Prompt = `SUB_AGENT_SYSTEM_PROMPT`
   - Returns the compiled agent

2. **`_execute_search_query(sub_agent, query: str) -> str`** async helper:
   - Invokes sub_agent with the query as a HumanMessage
   - Returns the final message content as string
   - On exception: returns `"Search failed: {str(e)}"`

3. **`build_research_orchestrator(settings)`** factory function:
   - Creates a `ChatOpenAI` using `AnalysisSettings.model` (gemini-pro) — the EXPENSIVE model
   - Uses `openai_api_base="https://openrouter.ai/api/v1"`, `openai_api_key=settings.openrouter_api_key`, `model=settings.analysis.model`, `temperature=settings.analysis.temperature`, `max_tokens=settings.analysis.max_tokens`
   - Uses `.with_structured_output(ResearchOutput)` for structured response
   - Returns the LLM (not a full agent — orchestrator plans then synthesizes, does not call tools directly)

4. **`research_node(state: PipelineState) -> dict`** async function:
   - Per D-06: if `state.get("skip_signal")` is True, return `{}` immediately (skip already decided)
   - Gets settings via `get_stake_settings()`
   - Builds runner context from `state["enriched_runners"]` and `state["parsed_race"]` via `_build_runners_context(state)`
   - **Phase 1 — Planning:** Call orchestrator LLM with `ORCHESTRATOR_SYSTEM_PROMPT` + runner context. First call asks: "Analyze these runners and create a research plan. Return a JSON list of search queries you want executed. Format: [{\"query\": \"search string\", \"purpose\": \"what this finds\"}]". Use a simple Pydantic model `ResearchPlan` with `queries: list[SearchQuery]` where `SearchQuery` has `query: str` and `purpose: str`. Use `.with_structured_output(ResearchPlan)` for this planning call.
   - **Phase 2 — Execution:** Build sub-agent via `build_search_sub_agent(settings)`. For each query in the plan, call `_execute_search_query(sub_agent, query.query)`. Collect results as list of `{"query": q.query, "purpose": q.purpose, "result": result_str}`.
   - **Phase 3 — Synthesis:** Call orchestrator LLM again with `ORCHESTRATOR_SYSTEM_PROMPT` + original runner context + all search results. This time use `.with_structured_output(ResearchOutput)` to get the final structured output.
   - Returns `{"research_results": research_output.model_dump(), "research_error": None}`
   - On exception: returns `{"research_results": None, "research_error": str(e)}`

5. Helper **`_build_runners_context(state: PipelineState) -> str`**:
   - Builds a human-readable summary of the race and runners for the orchestrator
   - Includes: track, race number, distance, surface, each runner's name/number/jockey/trainer/form/odds
   - Format: clear sections so the agent can parse it
